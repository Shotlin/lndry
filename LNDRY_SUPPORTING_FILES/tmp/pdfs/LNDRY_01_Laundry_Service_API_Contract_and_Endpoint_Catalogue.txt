Document purpose
This document is the target REST and real-time API contract for implementing the LNDRY Phase 1 laundry-service marketplace on the reusable Bakaloo technical 
foundation. It is intended for backend, Flutter, dashboard, QA and project-management use. Endpoint names may be refined during implementation, but the capability, 
access control and business outcome must remain equivalent.
Important implementation status
The Bakaloo backend already contains reusable authentication, orders, payments, addresses, delivery, reviews, notifications, shops, staff, finance, reporting, uploads and 
operational infrastructure. The LNDRY-specific endpoints below are the agreed target surface; some are adaptations of existing endpoints and others are new.
Service-domain declaration
LNDRY is exclusively a laundry-service booking and fulfilment platform. Garment types such as shirts, trousers, sarees, blankets and shoes are customer-owned articles 
submitted to a vendor for cleaning or care. They are used only to calculate the service charge by piece, pair or kilogram. The MVP workflow is service selection, garment 
declaration, pickup scheduling, payment, collection, laundry processing and return delivery.
Service-domain vocabulary
Term Meaning in LNDRY
Laundry service A vendor-offered operation such as Wash & Fold, Wash & Iron, Dry Cleaning, Ironing or 
Shoe Care.
Garment type A customer-owned article or service-measurement category, such as Shirt, Saree, Blanket, 
Shoes or Regular Clothes by kilogram.
Service rate The vendor charge for processing a garment type, expressed per piece, per pair or per 
kilogram.
Service quotation A server-calculated estimated service charge based on selected service, garment quantities 
or estimated weight.
Laundry service order A booked service job covering pickup, vendor processing and return delivery of the 
customer's garments.
Pickup and delivery Collection and return handover of the customer's own garments.

1. API-wide conventions
Convention Required rule
Base URL /api/v1 for application APIs; /api/webhook for verified third-party callbacks.
Authentication Bearer JWT access token plus rotating refresh token. Dashboard uses HttpOnly secure 
cookies or equivalent secure token storage.
Role scopes CUSTOMER, VENDOR_APPLICANT, VENDOR_OWNER, VENDOR_STAFF, RIDER, 
ADMIN, FINANCE_ADMIN, SUPER_ADMIN.
Identifiers UUIDs for users, vendors, services, orders, assignments, payments and documents.
Date/time ISO 8601 UTC in API. UI renders India Standard Time unless platform configuration 
changes.
Money Integer paise in API and database. Never use floating-point currency calculations.
Pagination page, limit, total, total_pages. Default 20; hard maximum 100 unless export endpoint.
Idempotency Required on payment, order acceptance, refunds and critical state-changing retries.
Errors Stable code, message, field_errors, request_id and safe metadata. No stack traces in 
production.
Audit All vendor approval, price adjustment, assignment, OTP verification, admin override, refund 
and 2FA actions are auditable.
Caching Public categories may be cached. Authenticated, order, capacity and payment responses 
use no-store.
Versioning Breaking changes require /api/v2 or compatible deprecation period.

3. Order state and transition rules
PAYMENT_PENDING -> WAITING_VENDOR_CONFIRMATION -> VENDOR_ACCEPTED -> PICKUP_ASSIGNED -> GOING_FOR_PICKUP -> PICKUP_OTP_VERIFIED -> 
PICKED_UP -> RECEIVED_AT_VENDOR -> PROCESSING -> PACKED -> DELIVERY_ASSIGNED -> OUT_FOR_DELIVERY -> DELIVERY_OTP_VERIFIED -> DELIVERED
 Vendor processing detail may be RECEIVED, WASHING, DRYING, IRONING and PACKED; the customer sees grouped statuses such as At Partner and 
Processing.
 Alternative terminal states include PAYMENT_FAILED, VENDOR_REJECTED, AUTO_REJECTED, CUSTOMER_CANCELLED, ADMIN_CANCELLED and 
REFUNDED.
 No client application may directly write an arbitrary status string. The backend validates every transition and actor permission.

6. Access-control matrix
Role Allowed Explicit restriction
Customer Own profile, addresses, discovery, quote, payment, own orders, 
own OTP display, own reviews.
Cannot access vendor private settings, rider data, other 
customers or admin operations.
Vendor applicant Own application and resubmission. Cannot publish services or receive orders until approval.
Vendor owner Own profile, services, prices, capacity, employees, vendor 
orders, revenue and performance. Cannot approve applications or access other vendors.
Vendor staff Permission-limited service/order operations. Cannot change ownership, documents, approved radius or high-
risk settings unless granted.
Rider Only own assigned pickup/delivery jobs and permitted customer 
address/contact.
Cannot change price, service, vendor profile, documents or 
arbitrary order status.
Platform admin Vendor approvals, customers, orders, exceptions, reports and 
configuration under permission model. High-risk actions require recent 2FA step-up and audit reason.
7. Backend implementation rules
 Reuse the existing Fastify route -> controller -> service -> repository architecture.
 Generate and maintain Swagger/OpenAPI documentation from route schemas.
 Use PostgreSQL transactions and row-level or advisory locking for slot holds, order creation and final capacity decrement.
 Hash login, pickup, delivery and direct 2FA OTP values; never store plaintext OTPs after verification.
 Use Redis for OTP challenges, rate limits, short-lived slot holds, idempotency keys and job coordination.
 Use BullMQ for vendor auto-reject, notification fan-out, watermark processing, payment expiry and retryable background work.
 Use server-side Razorpay signature validation and idempotent webhook processing as payment source of truth.
 Use Cloudinary restricted assets or equivalent signed delivery for sensitive documents; KYC originals must never be public URLs.
